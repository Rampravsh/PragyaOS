# PragyaOS Monorepo Engineering Standards (Constitution)

This document serves as the master source of truth for backend, frontend, and shared monorepo engineering standards across PragyaOS. Every developer and AI coding assistant must adhere to these rules without exception.

---

## 1. Monorepo Rules

*   **Standardized React Version**: All apps and packages must be locked to React `^19.2.0` (with corresponding typescript definitions).
*   **Path Aliases**: All application imports must use absolute path mappings (e.g. `@/*`). Relative jumps like `../../` are prohibited in product code.
*   **Documentation Synchronization**:
    *   Any change to tokens/theme requires updating `DESIGN_SYSTEM.md`.
    *   Any change to package boundaries requires updating `docs/architecture/PACKAGE_OWNERSHIP.md`.
    *   Any change to base architecture requires updating `docs/architecture/FRONTEND_ARCHITECTURE.md`.
    *   Any change to workspace rules requires updating `.agents/AGENTS.md`.

---

## 2. Package Ownership Rules

All monorepo elements must adhere to the modular boundaries documented in [PACKAGE_OWNERSHIP.md](file:///g:/PragyaOS/docs/architecture/PACKAGE_OWNERSHIP.md):
1.  **Strict Import Direction**: Imports must flow in a single direction (`apps` -> `packages/ui` -> `packages/theme` -> helper packages). Reverse imports (e.g. `@pragyaos/ui` importing from `apps/web`) are strictly forbidden.
2.  **Stateless Primitives Only**: `@pragyaos/ui` must only contain primitive component wrappers (Button, Input, Badge, base Card, Modal, Tooltip). It must never contain business logic, domain types, or API query hooks.
3.  **Encapsulated Assets**: No raw SVGs should be copy-pasted into product views. Icons must be exported from `@pragyaos/icons`, and illustrations must reside in `@pragyaos/assets`.

---

## 3. Design System & Theme Standards

1.  **Single Source of Truth**: `@pragyaos/theme` owns **every visual decision** across PragyaOS. This includes colors, typography, spacing, sizing, radius, grid, containers, motion, elevation, opacity, blur, breakpoints, interaction states, focus states, and layering.
2.  **No Redefined Tokens**: No component, CSS file, or JS file may introduce a new design token or override existing sizes, colors, or transitions locally.
3.  **Tailwind Preset Compliance**: All web applications must inherit the Tailwind configuration preset (`pragyaPreset`). Custom local brand overrides in `tailwind.config.ts` are forbidden.
4.  **Tailwind Arbitrary Values**: Arbitrary styles (e.g. `rounded-[var(--radius-sm)]`) are banned. Standard spacing, colors, and border shapes must use native Tailwind class overrides generated by the preset.
    *   *Exception*: Arbitrary values (`[var(--...)]`) are allowed only for properties without Tailwind utilities (like custom grid template divisions, clip-paths, masks, or dynamic animation calculations).

## 4. Frontend Standards

### Architecture Stabilization Rules
*   **Preserve Existing APIs**: Preserve existing APIs and function signatures wherever possible to prevent regressions.
*   **Preserve Component Props**: Do not change or delete component props unless absolutely necessary.
*   **Preserve Exports**: Keep all existing barrel and default exports.
*   **Refactor Internally First**: Optimize and update component/hook implementations internally before suggesting breaking external modifications.
*   **Breaking Changes Log**: If a breaking change is unavoidable, it must be documented in `docs/architecture/ARCHITECTURE_DECISIONS.md`.

### Component Rules
1.  **Size Limits**: React component files must not exceed 250 lines of code.
2.  **Props Limits**: Component interfaces must not contain more than 6 custom parameters (excluding native HTML attributes).
3.  **Nesting Limits**: JSX structures must not go deeper than 3 nesting levels (extract sub-views if they exceed this limit).
4.  **Feature Template Isolation**: Every domain feature under `apps/web/src/features/[domain]` must strictly follow the standardized template structure (`api/`, `components/`, `compositions/`, `hooks/`, `schemas/`, `services/`, `store/`, `types/`, `utils/`, `constants/`, `index.ts`). Cross-feature imports are prohibited; features can only expose endpoints via their public index barrel.

### Accessibility (A11y)
1.  **Semantic HTML**: Elements must use appropriate HTML5 tags (`<nav>`, `<header>`, `<main>`, `<button>`) instead of simple div clicks.
2.  **Keyboard Control**: Interactive elements must support full keyboard controls (Tab, Space, Enter, Escape).
3.  **Focus Management**: High-visibility focus rings must be attached to all active elements (`ring-2 ring-ring`).

### Performance
1.  **Bundle Budgets**: Initial main bundle size must stay below 150KB (gzip). Lazy-loaded route chunks must stay below 50KB.
2.  **Dynamic Code Splitting**: Route entries and complex sub-features must be lazy-loaded using dynamic imports.

## 5. Testing Standards

1.  **Test Stack**: Use **Vitest** for unit/integration testing, **React Testing Library (RTL)** for visual component tests, and **Mock Service Worker (MSW)** for API mocking.
2.  **Accessibility Testing**: Run automated accessibility audits using **axe-core/jest-axe** inside integration flows.

---

## 6. Backend Engineering Standards

1.  **No Direct Database Access in Services**: The Prisma database client `prisma` must NEVER be imported or invoked directly inside any service or controller file. All database reads and writes must be wrapped inside dedicated repositories.
    *   *Exception*: Complex database transaction blocks coordinating multiple repositories belong in the domain service layer.
2.  **Lightweight Probe Exceptions**: All standard controllers must utilize `SuccessResponse` wrappers. Liveness/readiness health probes return raw text payloads.
3.  **HttpStatus Constants Only**: Do not use raw magic integers for HTTP status codes. Enforce usage of the `HttpStatus` enum imported from `@pragyaos/constants`.
4.  **Zod Schema Validation**: Inputs must be parsed and validated at the controller entrypoint using Zod schemas.
5.  **Swagger Documentation**: Every route file must export Swagger/OpenAPI documentation decorators defining requests, responses, and parameters.
6.  **Security & RBAC Controls**: Endpoint routers must be guarded by auth checks (`requireAuth`) and role-based permissions gates.

## 7. AI Implementation Rules & Checklist

### Strict Duplication Ban
AI assistants **must never** create duplicate:
*   Components
*   Hooks
*   Icons or SVGs
*   Tokens
*   Utilities or Formatting Helpers
*   Layouts
*   Animations or Spring Configurations
*   CSS Custom Variables

### Definition of Done Validation
When acting as an AI assistant, you must check every implementation against the [FRONTEND_DEFINITION_OF_DONE.md](file:///g:/PragyaOS/docs/architecture/FRONTEND_DEFINITION_OF_DONE.md) checklist:
- [ ] Builds successfully without compiler/types warnings.
- [ ] No TypeScript errors (`tsc --noEmit` returns 0).
- [ ] No ESLint violations.
- [ ] Zero hardcoded colors or spacing properties.
- [ ] Strictly follows the package dependency direction rules.
- [ ] Accessible (keyboard navigable, focus rings, semantic tags).
- [ ] Responsive and supports layout grid margins.
- [ ] Fully documented if architecture patterns changed.
